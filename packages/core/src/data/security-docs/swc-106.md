# Unprotected Selfdestruct (SWC-106)

**Severity:** high
**Slither Check:** `suicidal`

## Description

The `selfdestruct` instruction in Solidity permanently destroys a smart contract, removing its code and storage from the blockchain, and sending any remaining Ether to a specified address. When a `selfdestruct` call is not protected by access control, any caller can destroy the contract at any time, irreversibly removing all functionality and sending all stored Ether to an address they control.

The `selfdestruct` operation is particularly dangerous because it is irreversible. Once executed, the contract's code is removed from the blockchain state. All functions become unavailable, all stored data is permanently deleted, and any tokens held by the contract (beyond Ether) are effectively lost forever since there is no contract code left to handle their withdrawal. Even if the contract is re-deployed at the same address using `CREATE2`, the storage would be empty.

Beyond direct fund loss, an unprotected `selfdestruct` can be used as a denial-of-service attack. If a contract system relies on a specific contract address (e.g., a registry or a proxy pattern), destroying that contract can break the entire system. Users who have approved tokens for the destroyed contract would need to re-approve for a new deployment, and any references to the old address become invalid.

## Vulnerable Code

```solidity
function destroy() external {
    selfdestruct(payable(msg.sender));
}
```

## Fixed Code

```solidity
function destroy() external onlyOwner {
    selfdestruct(payable(msg.sender));
}
```

## Real-World Impact

The **Parity Wallet Hack of November 2017** (a separate incident from the July hack) involved an unprotected `selfdestruct`. A user accidentally triggered the `kill()` function on the Parity wallet library contract, which had no access control. The `selfdestruct` destroyed the library contract that all Parity multi-sig wallets depended on for their logic. This permanently froze over 500,000 ETH (worth ~$150 million at the time) across hundreds of wallets, as they could no longer execute any functions without the library code.

While this was triggered accidentally, the same vulnerability could be exploited maliciously. An attacker could destroy a critical contract in a DeFi protocol, freezing all user funds that depend on it. For AI agent contracts, an attacker could destroy the agent entirely, permanently locking all user funds and tokens held by the contract.

It is worth noting that the Ethereum community has discussed removing or neutering `selfdestruct` in future protocol upgrades (EIP-4758 and EIP-6780). As of the Dencun upgrade (EIP-6780), `selfdestruct` only transfers Ether but no longer destroys contract code in most cases. However, contracts deployed before this change or using specific patterns may still be affected.

## How TrustLayer Detects This

TrustLayer's Slither scanner uses the `suicidal` detector to identify contracts that contain `selfdestruct` or `suicide` calls that can be triggered by any address. The detector checks whether the function containing the selfdestruct has proper access control.

The Permission Mapper flags this as a **self_destruct** capability, which is one of the most critical negative permissions in the TrustLayer scoring model. A contract that can be destroyed by anyone is immediately flagged as extremely dangerous. This contributes to the trust score through the Slither weight (30%) and Permissions weight (25%), and the presence of unprotected selfdestruct typically results in a severe trust score penalty (often pushing the score to D or F range).
