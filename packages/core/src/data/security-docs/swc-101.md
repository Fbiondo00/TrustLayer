# Integer Overflow/Underflow (SWC-101)

**Severity:** high
**Slither Check:** `integer-overflow`

## Description

Integer overflow and underflow occur when an arithmetic operation produces a result that is outside the range of the integer type used to store it. In Solidity, `uint256` wraps around: if you add 1 to `2^256 - 1`, the result becomes 0 (overflow). If you subtract 1 from 0, the result becomes `2^256 - 1` (underflow). Before Solidity 0.8.0, these operations did not revert; they silently produced incorrect results.

This vulnerability is particularly dangerous because it can bypass critical checks. For example, a balance check `require(balance >= amount)` can be bypassed if `amount` is crafted to cause an underflow, making the comparison evaluate incorrectly. Similarly, an attacker could mint tokens by causing an overflow in the total supply, or bypass rate limits by underflowing a counter.

Since Solidity 0.8.0, built-in overflow checks are included by default, which revert the transaction on overflow or underflow. However, developers can opt out using the `unchecked {}` block, which reintroduces the vulnerability. Additionally, older contracts compiled with Solidity <0.8.0 remain vulnerable. Many deployed contracts still use older versions, and even new contracts sometimes use `unchecked` for gas optimization without fully understanding the safety implications.

## Vulnerable Code

```solidity
function transfer(address to, uint256 amount) external {
    balances[msg.sender] -= amount;
    balances[to] += amount;
}
```

## Fixed Code

```solidity
using SafeMath for uint256;

function transfer(address to, uint256 amount) external {
    balances[msg.sender] = balances[msg.sender].sub(amount);
    balances[to] = balances[to].add(amount);
}
```

## Real-World Impact

The **BeautyChain (BEC) token** exploit in April 2018 is a classic example. The token's `batchTransfer` function multiplied the transfer amount by the number of recipients without checking for overflow. An attacker passed extremely large values that caused the multiplication to overflow to zero, allowing the attacker to mint tokens out of thin air and crash the token's value.

Another notable case was the **PoWHCoin (Proof of Weak Hands)** exploit, where an underflow in the token's fee calculation allowed an attacker to steal a significant portion of the contract's Ether. The **4chan /biz/ token** suffered a similar fate when an overflow in the reward calculation let attackers claim absurd amounts of tokens.

For AI agent contracts, an integer overflow could allow an attacker to manipulate the agent's internal accounting. For example, if an AI agent tracks user deposits and an overflow occurs in the deposit accounting, users could lose funds or an attacker could withdraw more than they deposited. If the agent manages token allowances or spending limits, an overflow could bypass those limits entirely.

## How TrustLayer Detects This

TrustLayer's Slither scanner uses the `integer-overflow` detector to identify arithmetic operations that lack overflow protection. This is especially important for contracts compiled with Solidity versions below 0.8.0 or those using `unchecked {}` blocks. Slither traces all arithmetic operations and checks whether they are protected by SafeMath or built-in overflow checks.

The Permission Mapper flags this as a potential **arithmetic_manipulation** risk, noting functions where unchecked math could lead to incorrect state changes. This contributes to the trust score through the Slither weight (30%) and Permissions weight (25%). Contracts using Solidity 0.8+ without `unchecked` blocks receive a clean bill of health on this check, while older contracts or those with unchecked blocks are flagged accordingly.
