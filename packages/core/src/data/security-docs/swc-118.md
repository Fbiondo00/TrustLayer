# Incorrect Constructor Name (SWC-118)

**Severity:** medium
**Slither Check:** (informational — detected by `naming-convention`)

## Description

In Solidity versions before 0.8.0, constructors were defined as functions with the same name as the contract. For example, a contract named `Token` would have a constructor function `function Token()`. If the constructor name was misspelled or didn't exactly match the contract name, Solidity would silently treat it as a regular public function instead of a constructor.

This means the "constructor" would be callable by anyone after deployment, allowing attackers to re-initialize the contract and change critical state variables like `owner`.

Solidity 0.8.0 removed this pattern entirely, replacing it with the `constructor()` keyword. However, many deployed contracts still use pre-0.8.0 syntax, and some developers copy patterns from older tutorials without realizing the risk.

This vulnerability is also relevant for upgradeable contracts that use `initialize()` instead of `constructor()`. If the initializer function name doesn't match the expected pattern, it might not be properly protected.

## Vulnerable Code

```solidity
// VULNERABLE: Pre-0.8.0 constructor name mismatch
pragma solidity ^0.7.0;

contract OwnedToken {
    address public owner;
    mapping(address => uint256) public balances;

    // VULNERABILITY: Typo in constructor name — "tokens" instead of "OwnedToken"
    // This is NOT a constructor — it's a regular public function!
    function tokens() public {
        owner = msg.sender;
    }

    // The REAL constructor is missing, so owner is address(0)
    // Anyone can call tokens() to become owner
    function mint(address to, uint256 amount) public {
        require(msg.sender == owner, "Not owner");
        balances[to] += amount;
    }
}

// Another common variant: contract renamed but constructor not updated
contract SafeVault {
    address public admin;

    // VULNERABILITY: Contract was renamed from "Vault" to "SafeVault"
    // but the constructor name was not updated
    function Vault() public {
        admin = msg.sender; // This is a regular function, not a constructor!
    }
}
```

## Fixed Code

```solidity
// SECURE: Use the constructor keyword (Solidity 0.8.x)
pragma solidity 0.8.28;

import "@openzeppelin/contracts/access/Ownable.sol";

contract SecureToken is Ownable {
    mapping(address => uint256) public balances;

    // SECURE: constructor keyword is unambiguous — no name matching needed
    constructor() {
        // owner is set by Ownable constructor
    }

    function mint(address to, uint256 amount) external onlyOwner {
        balances[to] += amount;
    }

    // For upgradeable contracts, use OpenZeppelin's Initializable
    // which prevents re-initialization via the initializer modifier
}

// SECURE: Upgradeable pattern with initializer guard
import "@openzeppelin/contracts-upgradeable/proxy/utils/Initializable.sol";
import "@openzeppelin/contracts-upgradeable/access/OwnableUpgradeable.sol";

contract SecureVaultV1 is Initializable, OwnableUpgradeable {
    mapping(address => uint256) public balances;

    /// @custom:oz-upgrades-unsafe-allow constructor
    constructor() {
        _disableInitializers(); // Prevents implementation from being initialized
    }

    // SECURE: initializer modifier ensures this runs exactly once
    function initialize() external initializer {
        __Ownable_init(msg.sender);
    }

    function deposit() external payable {
        balances[msg.sender] += msg.value;
    }
}
```

## Real-World Impact

**Rubixi (2016)** — The most famous case of constructor name mismatch. The Rubixi pyramid contract was renamed from `DynamicPyramid` to `Rubixi`, but the constructor function was left as `function DynamicPyramid()`. Since it didn't match the contract name, anyone could call `DynamicPyramid()` to become the contract owner and drain funds.

**Multiple Pre-0.8.0 Contracts** — Numerous contracts deployed before Solidity 0.8.0 had constructor name mismatches. These range from harmless (constructor was empty anyway) to critical (constructor set ownership and access control). Blockchain analysis has identified dozens of deployed contracts with this vulnerability.

**Parity Wallet Hack #1 (July 2017)** — While the primary issue was unprotected functions, the initWallet function was effectively a constructor substitute that could be called by anyone, similar to the constructor name mismatch pattern.

For AI agent contracts, this vulnerability is less relevant for new deployments (Solidity 0.8.x uses the `constructor` keyword). However, TrustLayer should still check for this pattern when analyzing older contracts or when the contract uses the upgradeable pattern with `initialize()` functions that lack proper guards.

## How TrustLayer Detects This

TrustLayer identifies incorrect constructor names through:

- **Slither (Step 3, 30% weight)** detects functions with names that match common constructor-like patterns but aren't declared with the `constructor` keyword. The `naming-convention` detector flags these as informational findings.

- **Permission Mapper (Step 5, 25% weight)** checks for initialization patterns. If the contract has a public function that sets ownership or critical state without access control, it's flagged as a potential constructor mismatch or unprotected initializer.

- **AI Analysis (Step 7, 10% weight)** evaluates the initialization architecture — detecting whether the contract uses `constructor()`, an older named-constructor pattern, or an upgradeable `initialize()` pattern. It flags initialization functions that lack proper guards (initializer modifier, access control).
