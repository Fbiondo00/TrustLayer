# Shadowing State Variables (SWC-119)

**Severity:** low
**Slither Check:** `shadowing-state`

## Description

State variable shadowing occurs when a derived contract declares a state variable with the same name as a state variable in a base contract. The derived variable does not override the base variable — instead, both exist in storage independently. When code in the derived contract references the variable name, it accesses the derived version, while code in the base contract still accesses the base version. This can lead to subtle and hard-to-detect bugs.

The problem is particularly insidious because the code appears to work correctly in simple tests but can produce unexpected behavior in complex scenarios. A developer reading the derived contract might assume that modifying `count` also updates the base contract's `count`, but this is not the case. The two variables occupy different storage slots and are completely independent.

In Solidity, state variables cannot be overridden in the same way as functions (which use the `virtual`/`override` keywords). Shadowing is always a bug or code smell — there is no legitimate reason to shadow a state variable. If a derived contract needs a different value for a state variable, the base contract should use a virtual function to expose the value, which the derived contract can override.

## Vulnerable Code

```solidity
contract Base {
    uint256 public count = 10;
}

contract Derived is Base {
    uint256 public count = 20;
}
```

## Fixed Code

```solidity
contract Base {
    uint256 public baseCount = 10;
}

contract Derived is Base {
    uint256 public derivedCount = 20;
}
```

## Real-World Impact

While no major exploit has been attributed solely to state variable shadowing, it has contributed to subtle bugs in complex contract systems. The risk is primarily a code correctness issue that can lead to unexpected behavior in production.

For example, consider a base contract that uses a `owner` variable for access control, and a derived contract that accidentally declares its own `owner` variable. Any access control checks in the base contract would use the base's `owner`, while code in the derived contract that references `owner` would use the derived's `owner`. This mismatch could effectively bypass access control or cause authorization confusion.

In proxy patterns and upgradeable contracts, state variable shadowing can cause storage layout collisions that corrupt critical data. The OpenZeppelin upgradeable contracts library explicitly warns against shadowing state variables because the storage layout must be consistent between implementation versions.

For AI agent contracts, shadowing could cause the agent to use incorrect configuration values, bypass intended limits, or make decisions based on the wrong data. For example, if a base contract defines `maxTradeSize` and a derived contract shadows it with a different value, risk management checks in the base contract would use the original limit while the derived contract's logic uses the new one, potentially allowing trades that exceed intended limits.

## How TrustLayer Detects This

TrustLayer's Slither scanner uses the `shadowing-state` detector to identify state variables in derived contracts that have the same name as state variables in any of their base contracts. The detector traverses the inheritance hierarchy and compares variable names across all levels.

This is classified as a low-severity finding, contributing a small penalty to the trust score through the Slither weight (30%). While not immediately exploitable, shadowing indicates poor coding practices that could mask more serious issues. TrustLayer's AI analysis flags this as a code quality concern and recommends renaming variables or refactoring the inheritance hierarchy.
