# Delegatecall to Untrusted Callee (SWC-112)

**Severity:** high
**Slither Check:** `controlled-delegatecall`

## Description

The `delegatecall` opcode executes code from another contract but in the context of the calling contract. This means the called code can read and write the calling contract's storage, access `msg.sender` and `msg.value` as if it were the original caller, and essentially take full control of the contract's state. When the target of a `delegatecall` is user-controlled, an attacker can execute arbitrary code that modifies any storage variable in the contract.

This vulnerability is particularly severe in proxy patterns, where a proxy contract delegates all calls to an implementation contract. If the implementation address can be changed by an untrusted party, the proxy's entire state — including ownership, balances, and access control — can be compromised. The attacker can deploy a malicious contract and point the proxy at it, then use the malicious contract to drain funds or take ownership.

Even outside of proxy patterns, `delegatecall` to untrusted addresses is extremely dangerous. A `delegatecall` preserves the storage layout of the calling contract, so the called code can directly modify any state variable by knowing (or guessing) its storage slot. This includes `owner`, `admin`, balance mappings, and any other critical state. The only safe way to use `delegatecall` is with targets that are either hardcoded or controlled by a trusted party through a governance process.

## Vulnerable Code

```solidity
function execute(address target, bytes calldata data) external {
    (bool ok, ) = target.delegatecall(data);
    require(ok);
}
```

## Fixed Code

```solidity
function execute(address target, bytes calldata data) external onlyOwner {
    require(whitelisted[target], "Target not whitelisted");
    (bool ok, ) = target.delegatecall(data);
    require(ok);
}
```

## Real-World Impact

The **Parity Wallet Hack of July 2017** involved a `delegatecall` vulnerability. The wallet contract delegated calls to a library contract for core functionality. An attacker exploited the fact that the library's initialization function could be called via `delegatecall` by anyone, allowing the attacker to overwrite the wallet's ownership storage slot and take control of all Parity multi-sig wallets.

The **Audius governance hack in July 2022** exploited a vulnerability in the protocol's proxy upgrade mechanism. An attacker was able to modify the implementation address of the governance contract, pointing it at a malicious contract that allowed them to mint and steal approximately $6 million worth of AUDIO tokens.

For AI agent contracts, a `delegatecall` vulnerability could allow an attacker to completely take over the agent. They could modify the agent's configuration, change its operator, redirect funds, or execute arbitrary operations in the agent's context. Since AI agents often have significant control over user funds and DeFi positions, a compromised agent could result in catastrophic losses for all users.

## How TrustLayer Detects This

TrustLayer's Slither scanner uses the `controlled-delegatecall` detector to identify functions that perform `delegatecall` to addresses that can be controlled by external callers. The detector performs taint analysis to trace whether the target address originates from user input (function parameters, calldata, or state variables writable by external functions).

The Permission Mapper flags this as a **delegatecall_unrestricted** risk, one of the most critical negative permissions. This contributes to the trust score through the Slither weight (30%) and Permissions weight (25%). The presence of user-controlled `delegatecall` almost always results in an F-level trust score due to the extreme risk it poses. TrustLayer's transaction history analysis may also flag suspicious patterns if the delegatecall target has been changed recently.
