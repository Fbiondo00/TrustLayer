# Unchecked Call Return Value (SWC-104)

**Severity:** medium
**Slither Check:** `unchecked-lowlevel`

## Description

In Solidity, low-level call operations such as `.call()`, `.delegatecall()`, and `.staticcall()` return a boolean value indicating whether the call succeeded or reverted. When the return value is not checked, a failed call silently proceeds without reverting the transaction. This means the contract continues executing as if the call had succeeded, potentially leading to incorrect state updates and lost funds.

The `.call()` function was introduced as the recommended way to send Ether and make external calls (replacing `.transfer()` and `.send()` which have a 2300 gas stipend limit). However, unlike `.transfer()` which reverts on failure, `.call()` only returns `false` on failure. If the developer does not explicitly check this return value, the transaction succeeds but the intended side effect (transferring Ether, calling a function) does not occur.

This vulnerability often appears in conjunction with other issues. For example, if a contract uses `.call()` to send Ether to a user and then updates state, the state update happens even though the Ether was never sent. In payment processing, withdrawal, or refund functions, this can lead to a situation where the contract records that a payment was made but the payment actually failed.

## Vulnerable Code

```solidity
function sendEther(address payable to, uint256 amount) external {
    to.call{value: amount}("");
}
```

## Fixed Code

```solidity
function sendEther(address payable to, uint256 amount) external {
    (bool ok, ) = to.call{value: amount}("");
    require(ok, "Transfer failed");
}
```

## Real-World Impact

While there is no single famous "billion-dollar" exploit attributed solely to unchecked return values, this vulnerability has been a contributing factor in many incidents. The **Lendf.me** hack in April 2020 (loss of ~$25 million) involved improperly handling return values from ERC20 token transfers, allowing an attacker to manipulate the protocol's internal balance accounting.

In the context of ERC20 tokens specifically, many tokens (like USDT) do not return a boolean from `transfer()` and `transferFrom()`, which means calling them with a standard interface that expects a return value can fail silently. This is why OpenZeppelin's `SafeERC20` library exists — it wraps token calls and properly handles both revert-based and return-value-based failure modes.

For AI agent contracts, an unchecked call could mean the agent believes it has executed a trade or transfer when it actually failed. This could lead to incorrect portfolio accounting, users being unable to withdraw their actual funds, or the agent making decisions based on incorrect state. If an agent is managing funds across multiple protocols, silent failures could cascade into significant accounting errors.

## How TrustLayer Detects This

TrustLayer's Slither scanner uses the `unchecked-lowlevel` detector to identify calls to `.call()`, `.delegatecall()`, and `.staticcall()` where the return value is not checked. The detector performs a data flow analysis to verify that the boolean return value is used in a `require()` statement or equivalent check.

The Permission Mapper flags functions with unchecked calls as having **unchecked_external_call** risk, noting that external calls without proper error handling can lead to silent failures. This contributes to the trust score through the Slither weight (30%) and Permissions weight (25%). The Dedaub token risk analysis also contributes by flagging tokens with non-standard return value behavior.
