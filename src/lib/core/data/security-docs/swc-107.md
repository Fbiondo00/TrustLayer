# Reentrancy (SWC-107)

**Severity:** high
**Slither Check:** `reentrancy-eth`

## Description

Reentrancy is one of the most well-known and devastating smart contract vulnerabilities. It occurs when an external call to an untrusted contract allows the called contract to re-enter the calling function before the first invocation has finished executing. This happens because the state updates (such as zeroing a balance) happen *after* the external call instead of *before* it.

The attack works by exploiting the order of operations in a function. When a contract sends Ether to an address using `.call()`, `.transfer()`, or `.send()`, the receiving contract's fallback or receive function is triggered. If the receiving contract is malicious, it can call back into the original function. Since the balance has not yet been updated, the function still "sees" the original balance and sends funds again. This loop continues until the contract is drained.

There are several variants of reentrancy beyond the classic single-function case. **Cross-function reentrancy** occurs when the re-entrant call targets a different function that shares state with the vulnerable one. **Cross-contract reentrancy** spans multiple contracts that share state dependencies. **Read-only reentrancy** affects view functions that are relied upon by external protocols, causing them to return stale or inconsistent data during the reentrant call.

## Vulnerable Code

```solidity
function withdraw() external {
    uint256 amount = balances[msg.sender];
    (bool ok, ) = msg.sender.call{value: amount}("");
    require(ok);
    balances[msg.sender] = 0;
}
```

## Fixed Code

```solidity
function withdraw() external nonReentrant {
    uint256 amount = balances[msg.sender];
    require(amount > 0);
    balances[msg.sender] = 0;
    (bool ok, ) = msg.sender.call{value: amount}("");
    require(ok);
}
```

## Real-World Impact

The most famous reentrancy attack was **The DAO hack of June 2016**, where an attacker exploited a reentrancy vulnerability in The DAO's `splitDAO` function. The attacker's fallback function recursively called `splitDAO` before the balance was updated, draining approximately 3.6 million ETH (worth ~$60 million at the time, billions today). This attack was so severe that it led to the Ethereum hard fork that created Ethereum Classic.

More recently, in 2022, the **Fei Protocol** lost ~$80 million due to a reentrancy vulnerability in its Rari Capital merger. The attacker used a flash loan combined with reentrancy to manipulate the protocol's internal accounting. In 2023, **Curve Finance** suffered a ~$70 million exploit due to a reentrancy bug in specific Vyper compiler versions that failed to properly implement the reentrancy guard.

For an AI crypto agent contract, a reentrancy vulnerability could allow an attacker to drain all user funds deposited into the agent, steal tokens held by the contract, or manipulate the agent's internal state to perform unauthorized operations. Even if the agent itself is not directly exploitable, if it interacts with other protocols on behalf of users, a reentrancy bug could cascade through the entire DeFi stack.

## How TrustLayer Detects This

TrustLayer's Slither scanner uses the `reentrancy-eth` detector to identify this pattern. Slither performs a static analysis of the control flow graph, tracking state variable reads and writes relative to external calls. If it finds an external call followed by a state write without an intervening reentrancy guard, it flags the vulnerability.

The Permission Mapper also flags this as a potential **reentrancy_risk** permission, noting functions that make external calls while modifying critical state. This contributes to the trust score through the Slither weight (30%) and Permissions weight (25%), making reentrancy one of the highest-impact vulnerabilities in the scoring model.
