# Block Values as Time Proxies (SWC-116)

**Severity:** low
**Slither Check:** `block-timestamp`

## Description

Solidity provides `block.timestamp` (alias `now` in older versions) as a way to reference the current block's timestamp. However, `block.timestamp` is not a precise clock — it is set by the miner (or validator) who produces the block, and they have limited ability to manipulate it within a small range (typically around 15 seconds). While this manipulation range is small, it can be significant for certain applications.

The vulnerability manifests when contracts rely on `block.timestamp` for critical timing decisions, such as lottery deadlines, auction endings, staking reward calculations, or time-locked withdrawals. A miner can slightly adjust the block timestamp to favor their own position, for example by moving the timestamp past a deadline to claim a reward or by delaying it to prevent another party from acting within a time window.

It is important to understand the realistic threat model. Miners cannot set `block.timestamp` to an arbitrary value — it must be greater than the timestamp of the parent block and is generally constrained by protocol rules and other miners' behavior. The typical manipulation range is about 15 seconds, though in practice it is usually much smaller. For most applications, this small window is acceptable. However, for applications where precise timing is critical (lotteries, high-frequency trading, or auctions with tight deadlines), even small manipulations can be exploited.

## Vulnerable Code

```solidity
function claimReward() external {
    require(block.timestamp >= unlockTime);
    _sendReward(msg.sender);
}
```

## Fixed Code

```solidity
function claimReward() external {
    require(block.timestamp >= unlockTime, "Not yet unlocked");
    _sendReward(msg.sender);
}
```

## Real-World Impact

No major exploit has been attributed solely to block timestamp manipulation, primarily because the manipulation range is small (approximately 15 seconds) and the cost of mining a block is high. However, the vulnerability has been exploited in combination with other attack vectors.

In **gaming and gambling contracts**, timestamp manipulation has been used to win lotteries or games of chance. A miner who is also a player could slightly adjust the block timestamp to ensure their ticket wins, especially in simple PRNG schemes that use `block.timestamp` or `blockhash` as a randomness source.

In **DeFi protocols**, timestamp manipulation can affect interest rate calculations, reward distributions, and TWAP (Time-Weighted Average Price) oracles. While the impact of a 15-second manipulation on a TWAP over hours or days is negligible, it can be meaningful for short-term operations like flash loans or intra-block arbitrage.

For AI agent contracts, if the agent uses `block.timestamp` for critical decisions (e.g., "execute this trade only during business hours" or "lock funds for exactly 24 hours"), a miner could slightly manipulate the timing. However, for most practical purposes, the 15-second window is acceptable, and the fixed code with explicit error messages provides better user experience without eliminating the inherent limitation.

## How TrustLayer Detects This

TrustLayer's Slither scanner uses the `block-timestamp` detector to identify usage of `block.timestamp` in the contract code. The detector flags instances where `block.timestamp` is used in conditions, especially in critical paths like fund transfers, access control, or reward calculations.

This is classified as a low-severity finding, contributing a small penalty to the trust score through the Slither weight (30%). The AI analysis step in TrustLayer's pipeline provides context-aware recommendations, explaining that while `block.timestamp` manipulation is possible, the small range (15 seconds) makes it acceptable for most non-critical timing purposes. For critical timing requirements, TrustLayer recommends using Chainlink or other oracle-based time feeds.
