# Weak Randomness (SWC-120)

**Severity:** medium
**Slither Check:** `weak-randomness`

## Description

Weak randomness in smart contracts refers to the use of predictable on-chain values as sources of randomness. Common sources include `blockhash`, `block.timestamp`, `block.number`, and `block.difficulty` (now `prevrandao` after The Merge). All of these values are either known in advance or can be influenced by miners/validators, making them unsuitable as randomness sources for any application where fairness is required.

The vulnerability is straightforward: any value that a miner or validator can predict or influence cannot serve as a fair random number. For `blockhash(block.number)`, the result is always zero because the current block's hash is not yet available. For `blockhash(block.number - 1)`, miners know the hash before they decide which transactions to include, allowing them to choose whether to include a transaction based on whether the resulting random number favors them. `block.timestamp` can be manipulated within a small range, and `prevrandao` (formerly `block.difficulty`) is determined by the protocol and is known to validators in advance.

Even for non-mining attackers, randomness based on on-chain values can often be predicted. Front-running bots monitor the mempool and can calculate the expected random outcome before submitting their own transaction. If the randomness source is `blockhash(block.number - 1) % N`, an attacker can compute the result for each possible block hash and only submit their transaction when the result favors them.

## Vulnerable Code

```solidity
function pickWinner() external {
    uint256 winner = uint256(blockhash(block.number - 1)) % players.length;
    address player = players[winner];
}
```

## Fixed Code

```solidity
// Use Chainlink VRF for secure randomness
function pickWinner() external {
    uint256 requestId = COORDINATOR.requestRandomWords(
        keyHash,
        subscriptionId,
        3,
        CALLBACK_GAS_LIMIT,
        1
    );
}
```

## Real-World Impact

**Fomo3D**, a popular Ethereum game in 2018, was famously exploited through weak randomness. The game used `block.timestamp` and other on-chain values to determine the winner. An attacker who was also a miner was able to manipulate the block timestamp to win the jackpot of approximately 10,469 ETH (worth ~$3 million at the time). While the attacker's exact method was debated, the game's reliance on predictable on-chain values made manipulation possible.

Numerous **lottery and gambling contracts** have been exploited due to weak randomness. In many cases, attackers did not need to be miners — they simply front-ran transactions by predicting the outcome based on pending block data. The **SmartBillions lottery** hack in 2017 exploited weak randomness to win the entire jackpot.

For AI agent contracts, weak randomness could be exploited in any feature that requires random selection: choosing which trading strategy to execute, selecting oracle sources, distributing rewards, or running lotteries. If an attacker can predict the random outcome, they can manipulate the agent's behavior for profit. For example, if an AI agent randomly selects between multiple DEX aggregators, an attacker could front-run the selection to ensure the agent trades on the DEX where the attacker has set up a sandwich attack.

## How TrustLayer Detects This

TrustLayer's Slither scanner uses the `weak-randomness` detector to identify usage of `blockhash`, `block.timestamp`, `block.number`, `block.difficulty`/`prevrandao`, or any combination of these values as a source of randomness. The detector traces data flow to find cases where these values are used in modular arithmetic or comparison operations that simulate random selection.

The Permission Mapper flags this as a **predictable_randomness** risk, noting that on-chain randomness can be predicted or manipulated. This contributes to the trust score through the Slither weight (30%) and Permissions weight (25%). TrustLayer's AI analysis recommends using Chainlink VRF or commit-reveal schemes for applications requiring fair randomness, and provides context-specific guidance based on how randomness is used in the contract.
