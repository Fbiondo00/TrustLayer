# Floating Pragma (SWC-103)

**Severity:** low
**Slither Check:** (informational — no direct detector)

## Description

A floating pragma is a Solidity source file that uses a version range instead of a specific compiler version. For example, `pragma solidity ^0.8.0;` allows compilation with any version from 0.8.0 to 0.8.x. This means the contract could be compiled with different compiler versions in different environments, potentially introducing behavioral differences.

While floating pragmas are convenient during development (they allow the code to compile with any compatible version), they create risk in production. Different compiler versions can produce different bytecode, have different optimization behaviors, or contain different bugs. A contract tested with one compiler version may behave differently when compiled with another.

The risk is primarily a code quality and operational concern rather than a direct exploit vector. However, it becomes serious when combined with other factors: if a compiler version has a known bug, a contract with a floating pragma could accidentally be deployed with that vulnerable version.

## Vulnerable Code

```solidity
// VULNERABLE: Floating pragma allows any 0.8.x compiler
pragma solidity ^0.8.0;

contract FloatingPragma {
    address public owner;

    constructor() {
        owner = msg.sender;
    }

    // This contract could be compiled with 0.8.0, 0.8.20, or 0.8.28
    // Each version produces different bytecode
    // 0.8.0 doesn't have some gas optimizations available in 0.8.28
    // 0.8.18 introduced PUSH0 which breaks compatibility with some chains
}
```

## Fixed Code

```solidity
// FIXED: Pinned pragma — exact compiler version
pragma solidity 0.8.28; // Specific version, no range

contract PinnedPragma {
    address public owner;

    constructor() {
        owner = msg.sender;
    }

    // This contract is ALWAYS compiled with 0.8.28
    // Bytecode is deterministic and reproducible
    // Testing matches deployment exactly
}
```

## Real-World Impact

The floating pragma issue contributed to the **Curve Finance incident (July 2023)**. While the primary cause was a Vyper compiler bug, the use of floating pragmas in some affected contracts meant that the specific compiler version used for deployment was not explicitly documented, making it harder to quickly identify which contracts were affected.

More broadly, **contracts deployed on chains that don't support the PUSH0 opcode** (introduced in Solidity 0.8.20, Shanghai EVM version) failed when compiled with 0.8.20+ using a floating pragma. This affected deployments on BSC, Avalanche, and other EVM-compatible chains that hadn't upgraded their EVM version.

For AI agent contracts, floating pragmas are an indicator of development rigor. A contract with a pinned pragma shows that the developer considered compilation determinism important. A floating pragma suggests the contract may not have undergone thorough deployment preparation.

## How TrustLayer Detects This

TrustLayer identifies floating pragmas through:

- **AI Analysis (Step 7, 10% weight)** parses the contract source code (or decompiled output) and flags floating pragmas as an informational finding. While not a critical vulnerability, it contributes to the overall code quality assessment.

- **Dedaub Decompiler (Step 2)** — When analyzing bytecode without source, the decompiler can sometimes identify the compiler version from metadata embedded in the bytecode.

The impact on the trust score is minor — floating pragmas contribute to the AI analysis weight (10%) and typically result in an informational note rather than a significant score reduction.
