# Unprotected Ether Withdrawal (SWC-105)

**Severity:** high
**Slither Check:** `missing-access-control`

## Description

An unprotected Ether withdrawal occurs when a smart contract function allows any caller to withdraw Ether from the contract without proper authorization checks. This is a critical access control failure where the contract fails to verify that the caller has the right to access the funds being withdrawn.

The vulnerability typically manifests in one of several ways: a withdrawal function that lacks an `onlyOwner` or similar modifier, a function that sends Ether to `msg.sender` without verifying their identity or balance, or a function that exposes `selfdestruct` without access control. In all cases, the result is the same — anyone can call the function and drain the contract's Ether balance.

This vulnerability is particularly dangerous because it is trivial to exploit. Unlike reentrancy which requires careful crafting of a malicious contract, an unprotected withdrawal can be exploited by simply calling the vulnerable function from any Ethereum account. The attack requires no special tools — a simple transaction from MetaMask or a basic web3 script is sufficient. The gas cost of the attack is minimal compared to the potential loss.

## Vulnerable Code

```solidity
function withdraw() external {
    payable(msg.sender).transfer(address(this).balance);
}
```

## Fixed Code

```solidity
function withdraw() external onlyOwner {
    payable(msg.sender).transfer(address(this).balance);
}
```

## Real-World Impact

The **Parity Wallet Hack of July 2017** is one of the most famous examples of unprotected function access. An attacker exploited a vulnerability in the Parity multi-sig wallet's `initWallet` function, which had no access control. The attacker called `initWallet` to take ownership of the wallet and then drained approximately 150,000 ETH (worth ~$30 million at the time) from multiple wallets.

The **BlackHat Ethereum incident** in 2023 exploited an unprotected withdrawal function in a yield aggregator, allowing the attacker to steal ~$1.2 million in user deposits. The attacker simply called a public function that was intended to be internal, draining the contract's entire balance.

For AI agent contracts, an unprotected withdrawal is catastrophic. If an AI agent manages user funds and has a public withdrawal function without access control, anyone can drain all user deposits instantly. Even if the agent only manages its own operational funds, losing those funds could prevent the agent from executing trades, paying for oracle services, or operating properly.

## How TrustLayer Detects This

TrustLayer's Slither scanner uses the `missing-access-control` detector to identify functions that perform sensitive operations (like Ether withdrawals) without proper authorization checks. The detector analyzes each function to determine whether it has access control modifiers and whether it performs operations that should be restricted.

The Permission Mapper specifically flags functions that can **transfer_unlimited** or **withdraw_without_auth** as critical negative permissions. This is one of the highest-priority checks in the TrustLayer pipeline, contributing significantly to the trust score through both the Slither weight (30%) and the Permissions weight (25%). Contracts with unprotected withdrawals receive severe score penalties and are typically flagged as high-risk or dangerous.
