# DoS with Failed Call (SWC-113)

**Severity:** medium
**Slither Check:** `dos-unchecked-return-value`

## Description

Denial of Service with Failed Call occurs when a contract iterates over a list of addresses and performs an operation (typically an Ether transfer) that can fail for any single address in the list. When one transfer fails, the entire transaction reverts, blocking the operation for all addresses in the list. This creates a denial-of-service vulnerability where a single malicious or broken recipient can prevent all other recipients from receiving their payments.

This vulnerability commonly appears in batch payment functions, airdrop distributions, dividend payments, and any function that iterates over a dynamic array of addresses. The issue is compounded when using `.transfer()` or `.send()`, which have a 2300 gas stipend that can fail if the recipient has a non-trivial fallback function. Even with `.call()`, a single recipient can intentionally revert to block the entire operation.

The attack is straightforward: an attacker adds a contract address to the recipients list that always reverts when receiving Ether. This single address blocks all payments in the batch. For time-sensitive operations like claim windows or distribution rounds, this DoS can cause permanent loss for legitimate recipients who cannot claim their share before a deadline passes.

## Vulnerable Code

```solidity
function payAll() external {
    for (uint i = 0; i < recipients.length; i++) {
        payable(recipients[i]).transfer(amounts[i]);
    }
}
```

## Fixed Code

```solidity
function payAll() external {
    for (uint i = 0; i < recipients.length; i++) {
        (bool ok, ) = recipients[i].call{value: amounts[i]}("");
        if (!ok) {
            failedPayments.push(recipients[i]);
        }
    }
}
```

## Real-World Impact

The **King of the Ether** game suffered from a DoS vulnerability where a smart contract was designed to always revert when receiving Ether. When this contract became the "king" (the highest bidder), no one could dethrone it because the payment to the previous king (the malicious contract) would always fail, reverting the entire transaction. This effectively froze the game.

In **batch airdrop and bounty distribution contracts**, DoS vulnerabilities have prevented thousands of legitimate recipients from claiming their tokens. In 2018, several ICO contracts that used batch distributions were rendered inoperable when attackers added revert-on-receive contracts to the distribution list.

For AI agent contracts, a DoS vulnerability could prevent the agent from processing user requests. If the agent iterates over a list of users (e.g., for distributing profits or executing batch operations), a single malicious user could block all operations. This could prevent timely trade execution, profit distribution, or fund withdrawals, potentially causing financial losses for all users.

TrustLayer's Slither scanner uses the `dos-unchecked-return-value` detector to identify loop patterns that make external calls which can fail individually, causing the entire transaction to revert. The detector analyzes control flow within loops to find patterns where `.transfer()`, `.send()`, or `.call()` failures would propagate to the transaction level.

The Permission Mapper flags this as a **dos_risk** pattern, noting that functions iterating over user-controlled lists with external calls are susceptible to denial-of-service.
