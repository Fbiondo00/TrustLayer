# DoS with Block Gas Limit (SWC-128)

**Severity:** medium
**Slither Check:** `calls-loop`, `reentrancy-benign`

## Description

Every Ethereum block has a gas limit (currently ~30 million gas). If a transaction requires more gas than the block limit, it cannot be included in a block and will always revert. A Denial of Service (DoS) attack with block gas limit exploits this by forcing a contract operation to iterate over an unbounded loop that eventually exceeds the block gas limit.

The most common pattern is a loop over a dynamically-sized array that an attacker can grow. For example, a contract that iterates over all token holders to distribute dividends will fail if the holder list grows too large. An attacker can register many addresses to inflate the array, making the distribution function permanently unusable.

This is particularly dangerous when the DoS'd function is required for the protocol to function (e.g., a withdrawal function, a claim function, or a governance execution function). If critical operations become permanently blocked, user funds can be locked forever.

**Key patterns:**

1. **Unbounded Loop Over User-Controlled Array** — An array that anyone can add to, which is then iterated over in a critical function.
2. **Payable Loop** — Iterating over an array and sending ETH to each address. ETH transfers to contracts trigger fallback functions, which can consume unpredictable gas.
3. **Nested Operations** — A loop that calls external contracts, where each call's gas cost is unpredictable.

## Vulnerable Code

```solidity
// VULNERABLE: DoS via unbounded loop
contract VulnerableAirdrop {
    address[] public recipients;
    mapping(address => bool) public isRecipient;

    // Anyone can register as a recipient — attacker adds many addresses
    function register() external {
        if (!isRecipient[msg.sender]) {
            recipients.push(msg.sender);
            isRecipient[msg.sender] = true;
        }
    }

    // VULNERABILITY: Iterates over unbounded array
    // If recipients.length > ~150, this exceeds block gas limit
    function distribute(uint256 amountPerRecipient) external payable {
        require(msg.value == amountPerRecipient * recipients.length, "Wrong amount");

        // If recipients grows large enough, this loop OOGs
        // and distribute() becomes permanently unusable
        for (uint256 i = 0; i < recipients.length; i++) {
            payable(recipients[i]).transfer(amountPerRecipient);
        }
    }
}

// VULNERABLE: DoS via payable loop (unpredictable gas per iteration)
contract VulnerableSplitter {
    address[] public payees;

    function addPayee(address payee) external {
        payees.push(payee);
    }

    function split() external payable {
        uint256 share = msg.value / payees.length;

        // VULNERABILITY: Each .transfer() to a contract consumes 2300+ gas
        // If any payee is a contract with expensive fallback, total gas exceeds limit
        for (uint256 i = 0; i < payees.length; i++) {
            payable(payees[i]).transfer(share);
        }
    }
}
```

## Fixed Code

```solidity
import "@openzeppelin/contracts/security/ReentrancyGuard.sol";

contract SecureAirdrop is ReentrancyGuard {
    mapping(address => uint256) public claimable;
    uint256 public totalClaimable;

    event Registered(address indexed user, uint256 amount);
    event Claimed(address indexed user, uint256 amount);

    // SECURE: Pull pattern — users claim individually (no loop)
    function register(uint256 amount) external {
        claimable[msg.sender] += amount;
        totalClaimable += amount;
        emit Registered(msg.sender, amount);
    }

    // SECURE: Each user claims independently — O(1) gas per claim
    function claim() external nonReentrant {
        uint256 amount = claimable[msg.sender];
        require(amount > 0, "Nothing to claim");

        // CEI pattern
        claimable[msg.sender] = 0;
        totalClaimable -= amount;

        (bool ok,) = msg.sender.call{value: amount}("");
        require(ok, "Transfer failed");

        emit Claimed(msg.sender, amount);
    }

    // SECURE: Paginated distribution if push pattern is needed
    function distributeBatch(address[] calldata users, uint256 amountPerUser) external nonReentrant {
        // Bounded by batch size passed in calldata, not by total user count
        for (uint256 i = 0; i < users.length; i++) {
            claimable[users[i]] += amountPerUser;
            // No ETH transfer in loop — just state updates
        }
    }
}

// SECURE: Pull-over-push pattern for splitting payments
contract SecureSplitter {
    mapping(address => uint256) public pendingWithdrawals;

    function deposit(address[] calldata payees) external payable {
        uint256 share = msg.value / payees.length;
        for (uint256 i = 0; i < payees.length; i++) {
            pendingWithdrawals[payees[i]] += share;
            // Only state updates in loop — no external calls
        }
    }

    // Each payee withdraws independently
    function withdraw() external {
        uint256 amount = pendingWithdrawals[msg.sender];
        require(amount > 0, "Nothing to withdraw");

        pendingWithdrawals[msg.sender] = 0;
        (bool ok,) = msg.sender.call{value: amount}("");
        require(ok);
    }
}
```

## Real-World Impact

**The DAO (2016)** — While the primary issue was reentrancy, the DAO's split mechanism also had a DoS vulnerability. The `splitDAO` function iterated over an array of proposals, and if this array grew too large, the function would exceed the block gas limit, preventing DAO token holders from splitting. A recursive "child DAO" attack was used to exacerbate this issue.

**King of Ether (2016)** — The contract maintained a list of all previous kings that was iterated during certain operations. As the list grew, the gas cost of operations increased until they exceeded the block gas limit, making the contract unusable.

**Various Airdrop Contracts** — Numerous airdrop contracts have become unusable because the distribution function iterates over all recipients. As the recipient list grows (especially when anyone can register), the gas cost eventually exceeds the block limit, permanently locking undistributed tokens.

For AI agent contracts, this vulnerability is relevant when agents process batches of operations. An agent that processes all pending trades in a single transaction could hit the gas limit if too many trades are queued. TrustLayer evaluates whether the agent uses bounded loops or pull patterns for batch operations.

## How TrustLayer Detects This

TrustLayer identifies DoS-with-block-gas-limit through:

- **Slither (Step 3, 30% weight)** uses the `calls-loop` detector to find external calls inside loops, and `reentrancy-benign` to identify loops with external calls that could fail. It also detects unbounded loops that depend on state variables that can grow without limit.

- **Permission Mapper (Step 5, 25% weight)** flags `batch_operations`, `loop_patterns`, and `mass_transfer` patterns. It evaluates whether loops are bounded, whether they contain external calls, and whether the iterated array can grow without limit.

- **AI Analysis (Step 7, 10% weight)** evaluates the contract's batch processing architecture — detecting pull-vs-push patterns, identifying critical functions that iterate over user-controlled arrays, and assessing whether pagination or individual claim patterns are used.
