# Transaction Order Dependence (SWC-114)

**Severity:** medium
**Slither Check:** `tx-origin`

## Description

Transaction Order Dependence (also known as front-running or sandwich attacks) occurs when a contract's behavior depends on the order in which transactions are included in a block. Since miners and validators control transaction ordering within a block, they can reorder, insert, or censor transactions to their advantage. Similarly, external observers can see pending transactions in the mempool and submit their own transactions with higher gas prices to be included first.

The vulnerability manifests when a contract's state changes based on the order of transactions. For example, if a function's behavior depends on the current state and that state can be changed between when a user submits a transaction and when it is mined, an attacker can front-run the user's transaction by submitting their own transaction with a higher gas price.

A related but distinct issue is the use of `tx.origin` for authentication, which is what the Slither `tx-origin` check specifically detects. The `tx.origin` global variable returns the original external account that started the transaction, not the immediate caller (`msg.sender`). This distinction is critical: if contract A calls contract B, and contract B checks `tx.origin`, it sees the user's address rather than contract A's address. A phishing attack can exploit this by tricking a user into calling a malicious contract that then calls the vulnerable contract.

## Vulnerable Code

```solidity
function withdraw() external {
    require(tx.origin == owner);
    payable(msg.sender).transfer(address(this).balance);
}
```

## Fixed Code

```solidity
function withdraw() external {
    require(msg.sender == owner, "Not owner");
    payable(msg.sender).transfer(address(this).balance);
}
```

## Real-World Impact

The **tx.origin phishing attack** is a well-documented attack vector. An attacker creates a malicious contract and tricks the owner of a vulnerable contract into calling it (e.g., by advertising it as an interesting DeFi opportunity). When the owner calls the malicious contract, it in turn calls the vulnerable contract's `withdraw()` function. Since `tx.origin` returns the original caller (the owner), the `require(tx.origin == owner)` check passes, and the malicious contract receives all the contract's Ether.

While specific high-profile losses directly attributed to `tx.origin` are less common than other vulnerabilities, the risk is well-understood in the security community. The Ethereum community has widely adopted the best practice of using `msg.sender` instead of `tx.origin` for authentication.

For AI agent contracts, using `tx.origin` for authentication means that if the agent's operator (owner) is tricked into interacting with a malicious contract, the attacker can call any function that uses `tx.origin` for auth. Given that AI agents often manage funds and execute trades, a successful tx.origin attack could result in complete loss of all funds under the agent's control.

## How TrustLayer Detects This

TrustLayer's Slither scanner uses the `tx-origin` detector to identify any use of `tx.origin` in the contract code. The detector flags all instances of `tx.origin` usage, regardless of whether they are used for authentication, as any use of `tx.origin` is generally considered a code smell and potential vulnerability.

The Permission Mapper flags `tx.origin` usage as an **auth_bypass** risk, noting that authentication through `tx.origin` can be circumvented through phishing attacks. This contributes to the trust score through the Slither weight (30%) and Permissions weight (25%). The AI analysis step provides specific guidance on replacing `tx.origin` with `msg.sender` or implementing proper multi-factor authentication.
