# Signature Malleability (SWC-117)

**Severity:** medium
**Slither Check:** `erc20-number-index`, informational from `unchecked-lowlevel`

## Description

ECDSA signatures in Ethereum consist of three components: `v` (recovery id), `r` (x-coordinate), and `s` (signature proof). The secp256k1 curve has a mathematical property that for any valid signature `(v, r, s)`, there exists a second valid signature `(v', r, s')` where `s' = n - s` (where `n` is the curve order). This means an attacker who observes a valid signature can create a different valid signature for the same message without knowing the private key.

If a contract checks whether a signature has been used by storing `(r, s)` or just the hash, the attacker can replay the same authorization with the malleable signature variant. This is signature malleability — the ability to create a different but equally valid signature for the same message.

The EIP-2 fork (Homestead) made non-canonical `s` values (`s > n/2`) invalid at the transaction level, but contracts that use `ecrecover` directly are still vulnerable because `ecrecover` accepts both canonical and non-canonical `s` values.

## Vulnerable Code

```solidity
// VULNERABLE: Signature malleability in meta-transaction
contract VulnerableMetaTx {
    mapping(bytes32 => bool) public executed;

    function execute(
        address to,
        uint256 amount,
        uint256 nonce,
        bytes memory sig
    ) external {
        bytes32 hash = keccak256(abi.encodePacked(to, amount, nonce, address(this)));
        bytes32 ethSignedHash = keccak256(abi.encodePacked("\x19Ethereum Signed Message:\n32", hash));

        // VULNERABILITY: ecrecover returns address for BOTH (r,s) and (r, n-s)
        address signer = ecrecover(ethSignedHash, sig);
        require(signer != address(0), "Invalid signature");
        require(!executed[hash], "Already executed");

        // Mark as executed using the hash
        executed[hash] = true;

        // But the attacker can create a second signature with s' = n - s
        // which produces the SAME hash but was NOT marked as executed
        // Wait — actually same hash. The vulnerability is when checking sig components.

        payable(to).transfer(amount);
    }

    // VULNERABILITY: Checking signature by storing r only
    mapping(uint256 => bool) public usedNonces;

    function withdraw(uint256 amount, uint256 nonce, bytes32 r, bytes32 s, uint8 v) external {
        bytes32 hash = keccak256(abi.encodePacked(msg.sender, amount, nonce));
        address signer = ecrecover(hash, v, r, s);
        require(signer == owner, "Not authorized");

        // VULNERABILITY: Only checking nonce, not s-value
        // Attacker can replay with malleable (r, n-s) signature
        require(!usedNonces[nonce], "Nonce used");
        usedNonces[nonce] = true;

        payable(msg.sender).transfer(amount);
    }

    address public owner;
}
```

## Fixed Code

```solidity
import "@openzeppelin/contracts/utils/cryptography/ECDSA.sol";
import "@openzeppelin/contracts/utils/cryptography/MessageHashUtils.sol";

contract SecureMetaTx {
    using ECDSA for bytes32;
    using MessageHashUtils for bytes32;

    mapping(bytes32 => bool) public executed;

    function execute(
        address to,
        uint256 amount,
        uint256 nonce,
        bytes memory signature
    ) external {
        bytes32 hash = keccak256(abi.encodePacked(to, amount, nonce, address(this)));
        bytes32 ethSignedHash = hash.toEthSignedMessageHash();

        // SECURE: ECDSA.recover validates s-value is in lower half of curve order
        // This rejects malleable signatures with s > n/2
        address signer = ECDSA.recover(ethSignedHash, signature);
        require(signer != address(0), "Invalid signature");
        require(!executed[hash], "Already executed");

        executed[hash] = true;
        payable(to).transfer(amount);
    }

    // Alternative: manual canonical s-value check
    function withdrawSecure(
        uint256 amount,
        uint256 nonce,
        bytes32 r,
        bytes32 s,
        uint8 v
    ) external {
        bytes32 hash = keccak256(abi.encodePacked(msg.sender, amount, nonce));
        bytes32 ethSignedHash = keccak256(abi.encodePacked("\x19Ethereum Signed Message:\n32", hash));

        // Manual canonical s-value check
        require(uint256(s) <= 0x7FFFFFFFFFFFFFFFFFFFFFFFFFFFFFFF5D576E7357A4501DDFE92F46681B20A0, "Non-canonical s");

        address signer = ecrecover(ethSignedHash, v, r, s);
        require(signer == owner, "Not authorized");
        require(!usedNonces[nonce], "Nonce used");
        usedNonces[nonce] = true;

        payable(msg.sender).transfer(amount);
    }

    mapping(uint256 => bool) public usedNonces;
    address public owner;
}
```

## Real-World Impact

**Bitcoin's Mt. Gox (2014)** — While not an Ethereum exploit, the most famous signature malleability attack. Attackers repeatedly withdrew funds by submitting transactions with malleable signatures. The exchange saw the "original" transaction fail (already broadcast) but the malleable variant succeeded, leading to accounting confusion and repeated withdrawals.

**ERC-20 Permit Front-Running** — Some ERC-20 permit implementations were vulnerable to signature malleability, allowing attackers to front-run permit calls with malleable signatures to drain approved tokens.

**Various Meta-Transaction Relayers** — Multiple relayer services experienced issues where malleable signatures caused transactions to be processed twice. The relayer saw one signature as "new" even though the malleable counterpart had already been processed.

For AI agent contracts, signature malleability is relevant when agents use meta-transactions, gasless approvals (ERC-2612 permit), or any off-chain signature verification. An agent that processes signed messages without canonical s-value validation can be tricked into executing the same authorization twice.

## How TrustLayer Detects This

TrustLayer identifies signature malleability through:

- **Slither (Step 3, 30% weight)** traces data flow through `ecrecover` calls and checks whether the returned signer is used for authorization without canonical s-value validation. It also checks for direct use of `ecrecover` instead of OpenZeppelin's `ECDSA.recover`.

- **Permission Mapper (Step 5, 25% weight)** flags `signature_verification` patterns. It evaluates whether signatures are verified using OpenZeppelin's safe `ECDSA` library or raw `ecrecover`, and whether replay protection (nonces, hash storage) accounts for malleable variants.

- **AI Analysis (Step 7, 10% weight)** examines the signature verification architecture holistically, identifying meta-transaction patterns, permit implementations, and any off-chain signature usage that might be vulnerable to malleability.
