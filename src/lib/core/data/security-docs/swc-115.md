# Authorization through tx.origin (SWC-115)

**Severity:** medium
**Slither Check:** `tx-origin`

## Description

Authorization through `tx.origin` is a specific variant of the tx.origin vulnerability pattern (SWC-114) that focuses on using `tx.origin` for access control decisions. The `tx.origin` variable in Solidity returns the address of the externally-owned account (EOA) that originally initiated the transaction, traversing the entire call chain. This is fundamentally different from `msg.sender`, which returns the immediate caller.

The vulnerability arises because `tx.origin` can be spoofed through phishing. If an attacker convinces the contract owner to interact with a malicious contract (e.g., by promoting it as a lucrative DeFi opportunity or a free token claim), the malicious contract can then call the vulnerable contract. When the vulnerable contract checks `tx.origin`, it sees the owner's address (since the owner initiated the original transaction), not the malicious contract's address. The authorization check passes, and the attacker's contract can perform any action reserved for the owner.

This vulnerability is especially insidious because it requires user interaction (the owner must be tricked into calling the malicious contract), making it a social engineering attack combined with a technical vulnerability. However, the consequences can be just as severe as a purely technical exploit — once the owner calls the malicious contract, all funds protected by the `tx.origin` check are immediately at risk.

## Vulnerable Code

```solidity
function transfer(address to, uint256 amount) external {
    require(tx.origin == owner);
    _transfer(msg.sender, to, amount);
}
```

## Fixed Code

```solidity
function transfer(address to, uint256 amount) external {
    require(msg.sender == owner, "Unauthorized");
    _transfer(msg.sender, to, amount);
}
```

## Real-World Impact

The classic attack scenario involves a phishing website or social engineering campaign. An attacker creates a website that looks like a legitimate DeFi protocol and invites the target (who they know is the owner of a vulnerable contract) to claim a reward or participate in an exclusive event. When the target connects their wallet and clicks "Claim," the website triggers a transaction to a malicious contract. That contract immediately calls the vulnerable contract's protected function, and since `tx.origin` is the target's address, the authorization check passes.

In practice, this has been a persistent attack vector in the Ethereum ecosystem. Phishing attacks through fake airdrops, token claims, and NFT mints are common. While the specific use of `tx.origin` in the target contract makes the attack easier, even contracts that use `msg.sender` can be exploited if the user is tricked into signing a malicious transaction directly.

For AI agent contracts, the risk is amplified because AI agents often have significant control over user funds. If an agent's operator authenticates using `tx.origin`, a successful phishing attack on the operator could allow an attacker to take over the agent entirely, redirecting all user funds or manipulating the agent's trading strategy.

## How TrustLayer Detects This

TrustLayer's Slither scanner uses the `tx-origin` detector to identify any use of `tx.origin` in the contract code. The detector performs a pattern-matching analysis on the contract's AST to find all references to `tx.origin` and flags them regardless of the specific usage context.

The Permission Mapper flags this as an **insecure_auth** pattern, noting that authorization checks based on `tx.origin` can be bypassed through phishing. This contributes to the trust score through the Slither weight (30%) and Permissions weight (25%). TrustLayer's AI analysis provides detailed explanations of why `tx.origin` is dangerous and recommends replacing it with `msg.sender` or role-based access control patterns.
