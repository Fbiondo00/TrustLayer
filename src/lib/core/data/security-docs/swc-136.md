# Read-Only Reentrancy (SWC-136)

**Severity:** high
**Slither Check:** `reentrancy-eth`, `constant-function-state` (indirect)

## Description

Read-only reentrancy is a subtle variant of the standard reentrancy attack. Instead of the attacker re-entering a function that modifies state (which is caught by most reentrancy guards), the attacker exploits a `view` or `pure` function that reads stale state during an ongoing state transition.

The attack works as follows: a contract updates some state (e.g., exchange rate) and then makes an external call. During that external call, the attacker calls a `view` function on the contract (or a third-party contract that reads from it). The `view` function reads the partially-updated state — for example, it sees the new exchange rate but the old balance, or vice versa. The attacker uses this inconsistent view to exploit another protocol that relies on the `view` function's output.

This is particularly dangerous because `view` functions are typically considered safe and aren't protected by reentrancy guards. Cross-protocol read-only reentrancy (where the vulnerable `view` function is in one protocol but the exploiter is in a completely different protocol) has become a major attack vector in DeFi.

## Vulnerable Code

```solidity
// VULNERABLE: Read-only reentrancy in a vault contract
contract VulnerableVault {
    uint256 public totalAssets;
    uint256 public totalShares;
    mapping(address => uint256) public shares;

    // Exchange rate: assets per share
    function exchangeRate() public view returns (uint256) {
        if (totalShares == 0) return 1e18;
        return (totalAssets * 1e18) / totalShares;
    }

    // VULNERABILITY: State updated before external call
    function withdraw(uint256 amount) external {
        uint256 userShares = shares[msg.sender];
        require(userShares >= amount, "Insufficient shares");

        uint256 assets = (amount * exchangeRate()) / 1e18;
        require(assets <= totalAssets, "Insufficient liquidity");

        // Update shares FIRST (but not totalAssets!)
        shares[msg.sender] -= amount;
        totalShares -= amount;

        // VULNERABILITY: External call BEFORE totalAssets is updated
        // During this call, exchangeRate() returns a HIGHER rate
        // because totalShares decreased but totalAssets didn't
        (bool ok,) = msg.sender.call{value: assets}("");
        require(ok);

        // State update AFTER external call — stale read window
        totalAssets -= assets;
    }

    function deposit() external payable {
        uint256 sharesToMint = (msg.value * 1e18) / exchangeRate();
        shares[msg.sender] += sharesToMint;
        totalShares += sharesToMint;
        totalAssets += msg.value;
    }
}

// Attacker contract exploiting read-only reentrancy
contract Attacker {
    VulnerableVault vault;
    IDeFiProtocol lending; // Another protocol reading vault.exchangeRate()

    function attack() external payable {
        // Deposit to get shares
        vault.deposit{value: msg.value}();

        // Withdraw triggers the callback
        vault.withdraw(vault.shares(address(this)));

        // During the receive() callback:
        // - vault.totalShares is decreased
        // - vault.totalAssets is NOT yet decreased
        // - vault.exchangeRate() returns inflated value
        // - lending protocol uses this inflated rate to calculate collateral
    }

    receive() external payable {
        // Read-only reentrancy: call a DIFFERENT protocol
        // that reads vault.exchangeRate() (stale/inflated)
        uint256 inflatedValue = lending.getCollateralValue(address(this));
        lending.borrow(inflatedValue * 90 / 100); // Borrow against inflated collateral
    }
}
```

## Fixed Code

```solidity
import "@openzeppelin/contracts/security/ReentrancyGuard.sol";

contract SecureVault is ReentrancyGuard {
    uint256 public totalAssets;
    uint256 public totalShares;
    mapping(address => uint256) public shares;

    // Reentrancy lock for view functions too
    uint256 private _locked = 1;

    function exchangeRate() public view returns (uint256) {
        // SECURE: Check that no state-modifying function is in progress
        // This prevents read-only reentrancy
        require(_locked == 1, "Reentrant call");
        if (totalShares == 0) return 1e18;
        return (totalAssets * 1e18) / totalShares;
    }

    function withdraw(uint256 amount) external nonReentrant {
        uint256 userShares = shares[msg.sender];
        require(userShares >= amount, "Insufficient shares");

        uint256 assets = (amount * exchangeRate()) / 1e18;
        require(assets <= totalAssets, "Insufficient liquidity");

        // CEI: Update ALL state before external call
        shares[msg.sender] -= amount;
        totalShares -= amount;
        totalAssets -= assets; // SECURE: Updated BEFORE external call

        (bool ok,) = msg.sender.call{value: assets}("");
        require(ok, "Transfer failed");
    }

    function deposit() external payable nonReentrant {
        uint256 sharesToMint = (msg.value * 1e18) / exchangeRate();
        shares[msg.sender] += sharesToMint;
        totalShares += sharesToMint;
        totalAssets += msg.value;
    }
}

// Alternative: use a dedicated read-only reentrancy guard
contract ReadOnlyGuard {
    uint256 private _readOnlyLock = 1;

    modifier nonReentrantReadOnly() {
        require(_readOnlyLock == 1, "Read-only reentrancy");
        _readOnlyLock = 0;
        _;
        _readOnlyLock = 1;
    }

    // Call this at the start of view functions that other protocols rely on
    // function exchangeRate() public view nonReentrantReadOnly returns (uint256) { ... }
}
```

## Real-World Impact

**Curve Finance Read-Only Reentrancy (Multiple Exploits, 2023)** — After the Vyper compiler bug incident, attackers also exploited read-only reentrancy in Curve pools. The attacker called a function that updated the pool's virtual price, then during the callback, called a lending protocol that used `virtual_price()` as an oracle. The stale/inflated virtual price allowed the attacker to borrow more than their collateral was worth.

**Balancer ($23M, June 2023)** — An attacker exploited read-only reentrancy in Balancer pools. The attack involved calling `withdraw` on a Balancer pool, which updated some state but not the protocol fee snapshot. During the callback, the attacker interacted with a DeFi protocol that read the pool's stale rate, enabling them to drain funds.

**dYdX / MakerDAO Oracle Incidents** — Multiple incidents where protocols reading from other protocols' view functions received stale data during state transitions. While not always exploited for direct theft, they demonstrate the systemic risk of cross-protocol view function dependencies.

**Sentiment Protocol ($1M, April 2023)** — The attacker exploited read-only reentrancy by manipulating the exchange rate of a liquidity pool during a withdrawal, then using the inflated rate to borrow against it in the Sentiment lending protocol.

For AI agent contracts, read-only reentrancy is particularly relevant because agents often call `view` functions on external protocols to make decisions. If an agent reads a stale exchange rate from a protocol mid-state-transition, it could make a catastrophically wrong trading decision. TrustLayer evaluates whether the agent validates data freshness and has circuit breakers for anomalous values.

## How TrustLayer Detects This

TrustLayer identifies read-only reentrancy through:

- **Slither (Step 3, 30% weight)** uses `reentrancy-eth` to detect the pattern where state is not fully updated before an external call. It also detects view functions that read state which could be stale during an ongoing state transition.

- **Permission Mapper (Step 5, 25% weight)** flags `oracle_read`, `rate_query`, and `balance_view` patterns. It evaluates whether view functions are protected against reentrancy, whether state updates are complete before external calls, and whether the contract is used as an oracle by other protocols.

- **AI Analysis (Step 7, 10% weight)** examines the contract's state transition patterns holistically — detecting partial state updates, identifying view functions that read partially-updated state, and evaluating cross-protocol oracle dependencies.
