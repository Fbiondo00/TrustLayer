# Assert Violation (SWC-110)

**Severity:** medium
**Slither Check:** `incorrect-equality`

## Description

The `assert()` function in Solidity is designed to check for invariants — conditions that should never be false under any circumstances if the code is correct. It is not meant for input validation or checking user-supplied values. When `assert()` is used incorrectly (e.g., for input validation), it consumes all gas sent with the transaction upon failure (pre-Istanbul hard fork) and indicates a logic error rather than an expected failure case.

The proper use of `assert()` is limited to checking internal consistency: verifying that a variable is within an expected range after an operation, confirming that a state transition was executed correctly, or checking conditions that should never be violated if the contract code is bug-free. Using `assert()` for user input validation is a misuse that wastes gas and can cause denial-of-service when legitimate inputs trigger the assertion.

Since the Istanbul hard fork (EIP-2200), the gas cost of `assert()` failure was reduced to the remaining gas instead of all gas, but the semantic distinction remains important. A failing `assert()` indicates a bug in the contract code, while a failing `require()` indicates an invalid input or condition that the contract should handle gracefully with a descriptive error message.

## Vulnerable Code

```solidity
function setPrice(uint256 _price) external {
    assert(_price > 0);
    price = _price;
}
```

## Fixed Code

```solidity
function setPrice(uint256 _price) external {
    require(_price > 0, "Price must be positive");
    price = _price;
}
```

## Real-World Impact

While no single massive exploit is attributed directly to `assert()` misuse, it has caused real problems in production contracts. In the **Ethereum Name Service (ENS)** early versions, improper use of `assert()` in some paths could cause transactions to fail unexpectedly, preventing users from performing valid actions.

The more insidious risk is that `assert()` misuse can mask underlying logic errors. If a developer uses `assert()` to validate a condition that can actually be triggered by user input, they may not properly handle the failure case. The transaction simply reverts with no useful error message, making debugging difficult and potentially locking users out of functionality.

For AI agent contracts, improper use of `assert()` could cause the agent to become temporarily unusable if a valid user action triggers an assertion. This could prevent the agent from executing trades, processing withdrawals, or responding to user commands. In a time-sensitive trading context, a DoS caused by an assertion failure could result in significant financial losses due to missed opportunities or inability to close positions.

## How TrustLayer Detects This

TrustLayer's Slither scanner uses the `incorrect-equality` detector (and related checks) to identify misuse of `assert()` for conditions that should use `require()`. The detector looks for `assert()` calls that depend on function parameters or external inputs, as these should typically use `require()` instead.

The TrustLayer pipeline classifies this as a medium-severity finding, contributing to the trust score through the Slither weight (30%). While not as critical as reentrancy or access control issues, the presence of `assert()` misuse indicates poor coding practices that may correlate with other, more serious vulnerabilities. The AI analysis step in TrustLayer's pipeline specifically calls out coding quality issues to help users understand the overall risk profile.
