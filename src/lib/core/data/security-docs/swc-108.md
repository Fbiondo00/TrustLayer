# State Variable Default Visibility (SWC-108)

**Severity:** low
**Slither Check:** `state-visibility`

## Description

In Solidity, state variables that do not have an explicitly declared visibility specifier default to `internal`. While this is a safe default in terms of access, the lack of explicit visibility can lead to confusion, maintenance issues, and unintended behavior if a developer assumes the variable is `private` or `public` when it is not. More critically, some compilers or analysis tools may handle implicit visibility differently, and future Solidity versions may change defaults.

The primary risk is not an immediate vulnerability but rather a maintenance and code clarity issue. When visibility is implicit, future developers or auditors must know the default behavior to understand the contract's access patterns. If a variable is intended to be `private` but lacks the specifier, it will actually be `internal` and accessible by derived contracts. This could expose sensitive state that was meant to be encapsulated.

Additionally, in more complex contract systems, implicit visibility can interact dangerously with inheritance. An `internal` variable is accessible by all derived contracts, which may not be the developer's intention. If a derived contract is later made public or is interacted with by untrusted parties, the internal state could be indirectly exposed or manipulated.

## Vulnerable Code

```solidity
mapping(address => uint256) balances;
```

## Fixed Code

```solidity
mapping(address => uint256) private balances;
```

## Real-World Impact

While no major exploit has been directly attributed to default visibility, this issue contributes to a class of "code quality" vulnerabilities that can compound into real security issues. In large codebases or contracts that undergo multiple rounds of modification by different developers, implicit visibility can lead to incorrect assumptions about who can access certain state.

For example, if a developer adds a variable tracking admin addresses without specifying `private`, it becomes `internal`. A derived contract could read or modify this variable, potentially bypassing admin checks. In proxy patterns (common in upgradable contracts), the visibility of storage variables must be carefully managed to prevent storage collisions.

In the context of AI agent contracts, a state variable with implicit visibility could allow unexpected access to critical configuration data, such as API keys (stored as hashes), authorized caller lists, or fee parameters. While the direct risk is low, it contributes to an overall "code quality debt" that makes the contract harder to audit and more prone to future vulnerabilities.

## How TrustLayer Detects This

TrustLayer's Slither scanner uses the `state-visibility` detector to identify state variables that lack an explicit visibility specifier. The detector parses the contract's Abstract Syntax Tree (AST) and flags any state variable declaration that does not include `public`, `private`, `internal`, or `external` keywords.

This is classified as a low-severity finding, contributing a small penalty to the trust score through the Slither weight (30%). However, the presence of multiple low-severity findings can compound into a meaningful score reduction. TrustLayer's AI analysis step also notes code quality issues when explaining the overall trust assessment, helping users understand not just critical vulnerabilities but also overall code hygiene.
